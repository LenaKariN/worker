# 4. Стратегия решения

## 4.1. Архитектурные решения

### CLI/Typer
Реализовать приложение с консольным интерфейсом (CLI) с использованием библиотеки [Typer](https://pypi.org/project/typer-cli).  
Мотивация: [TC01](02_Constraints.md#21-технические-ограничения).

Реализация CLI должна быть эстетически привлекательной ([NFR02](01_Introduction_and_Goals.md#13-цели-качества)), 
а расширение функциональности выполняться по возможности легко ([NFR04](01_Introduction_and_Goals.md#13-цели-качества)).

### Основной способ хранения данных: in-memory
Простая реализация, основанная на динамических массивах.  
Мотивация: [TC02](02_Constraints.md#21-технические-ограничения).

### Диалоговый режим CLI
Обычно CLI имеет жизненный цикл `stateless` (завершается после выполнения каждой команды).
Однако, для демонстрации реализации ФТ (см. [1.2. Обзор требований](./01_Introduction_and_Goals.md#12-обзор-требований)) 
потребуются более сложные сценарии, например:
1. _Создать нового Сотрудника_
2. _Создать еще одного Сотрудника_
3. _Редактировать первого Сотрудника_
4. _Просмотреть список Сотрудников_

Для этого сценария необходимо либо сохранять данные между запросами в БД, либо сделать приложение `stateful`, 
тогда данные смогут сохраняться в оперативной памяти.  
Следовательно, на основе ограничения [TC02](./02_Constraints.md#21-технические-ограничения) 
CLI должно поддерживать диалоговый (REPL) режим: команды не завершают его работу и имеют общий контекст.

### Редактирование в режиме диалога
При редактировании выбранного объекта CLI должен предоставить возможность менять свойства объекта по отдельности, 
так как чаще всего меняется одно поле из множества.
Требуется сделать редактирование в диалоговом режиме:
объект отображается в табличном представлении, а затем пользователь выбирает нужное поле и вводит новое значение.

### Использование паттерна Command
Для поддержания слабой связанности кода и легкой управляемости будет использован паттерн Command.
Каждая функция системы будет содержаться в отдельной команде, 
а входные/выходные параметры команды будут динамически отображаться в CLI.

### Постраничная загрузка коллекций
Все коллекции элементов должны загружаться из хранилища постранично.
В UI такие коллекции так же отображаются постранично.

### Использовать Pydantic для валидации
Для поддержания логической целостности доменной модели требуется встроить в нее валидационные правила.
Необходимо использовать фреймворк Pydantic, так как он отлично подходит для этой задачи и позволит сэкономить время на разработку механизма валидации.

### Целочисленный идентификатор сущностей
При работе через CLI придется часто вводить ID объектов, поэтому он должен быть простым и коротким.
Лучший выбор - целое положительное число.

### Чистая архитектура (Ports & Adapters)
Применяется архитектурный стиль Clean Architecture (Hexagonal Architecture)
с разделением кода на три слоя: `domain`, `application`, `adapters`.
Бизнес-логика изолирована в доменном слое и не зависит от инфраструктуры.
Слой `application` содержит сценарии использования (use cases),
а слой `adapters` — входные (CLI) и выходные (in-memory storage, time, ID) адаптеры.

Мотивация: [NFR04](01_Introduction_and_Goals.md#13-цели-качества) — легкое расширение
функциональности без изменения существующего кода.
[NFR03](01_Introduction_and_Goals.md#13-цели-качества) — возможность вносить
точечные изменения в изолированных слоях.

### Domain-Driven Design
В доменном слое используются тактические паттерны DDD:
Entity, Value Object, Aggregate, Repository.
Value Objects (`Money`, `Effectivity`, `PersonName`, `Phone`) — неизменяемые объекты-значения,
инкапсулирующие валидацию.
Aggregate — корень агрегата (`Employee`, `PositionType`), обеспечивающий целостность данных.
Repository — абстрактный интерфейс доступа к хранилищу, реализованный в слое `adapters`.

Доменная модель основана на паттерне Accountability (M. Fowler, "Analysis Patterns"):
сущности `Party` (`Employee`, `OrganizationalUnit`) участвуют в отношениях ответственности,
что позволяет моделировать произвольную организационную структуру.

Мотивация: [NFR05](01_Introduction_and_Goals.md#13-цели-качества) — строгое поддержание
целостности данных через инкапсуляцию бизнес-правил в доменных объектах.
[OC02](02_Constraints.md#22-организационные-ограничения) — изучение аналитического
паттерна Accountability.

### Внедрение зависимостей (IoC)
Для управления зависимостями используется IoC-контейнер
[punq](https://pypi.org/project/punq).
Все инфраструктурные зависимости регистрируются в контейнере в Composition Root.
Команды получают зависимости через конструктор, что позволяет легко подменять
реализации для тестирования.

Мотивация: [NFR04](01_Introduction_and_Goals.md#13-цели-качества) — легкое расширение
функциональности. Новый use case достаточно создать и поместить в пакет `usecases` —
он будет автоматически обнаружен и зарегистрирован в CLI.
[NFR03](01_Introduction_and_Goals.md#13-цели-качества) — подмена зависимостей
для тестирования без изменения production-кода.

### Авто-обнаружение сценариев использования
Все команды автоматически обнаруживаются при запуске приложения и инициализируются на основе метаданных. 

Мотивация: [NFR04](01_Introduction_and_Goals.md#13-цели-качества) — добавление нового
сценария сводится к созданию одного файла с классом, без модификации CLI-адаптера.

### Мягкое удаление (Soft Delete)
Физическое удаление сущностей не предусмотрено. Вместо этого используется поле `is_deleted`.
При `is_deleted=True` запись считается удаленной и не возвращается в списках по умолчанию,
но остается в хранилище для обеспечения ссылочной целостности и аудита.
Параметр `include_deleted` позволяет при необходимости просматривать удаленные записи.

Мотивация: [FR01](01_Introduction_and_Goals.md#12-обзор-требований) — требование
индивидуального задания не предусматривает безвозвратное удаление.
[TC02](02_Constraints.md#21-технические-ограничения) — в in-memory хранилище
мягкое удаление реализуется тривиально переключением флага.

### Справочники (Reference Type)
Для хранения типовых справочных данных (должности, типы подразделений, типы отношений)
используется паттерн `ReferenceType` - базовый класс с полями `code`, `name`,
`description` и `effectivity`. Код справочного элемента — машиночитаемый уникальный
идентификатор (латиница, цифры, подчёркивание), имя — человекочитаемое представление.

Мотивация: [FR04](01_Introduction_and_Goals.md#12-обзор-требований) — управление
справочником позиций. Паттерн легко расширяется для FR05-FR06 через наследование
от `ReferenceType`.

### Rich для визуализации CLI
Для отрисовки таблиц и панелей в терминале используется библиотека
[Rich](https://pypi.org/project/rich). Таблицы применяются для списков (коллекций),
панели — для детального просмотра одной сущности.

Мотивация: [NFR02](01_Introduction_and_Goals.md#13-цели-качества) — эстетически привлекательный CLI.
[NFR01](01_Introduction_and_Goals.md#13-цели-качества) — табличное представление
данных интуитивно понятно пользователю.

### Интерактивное редактирование (FieldEditor)
Для диалогового редактирования используется паттерн FieldEditor — абстрактный
редактор одного поля сущности. Каждое поле представлено отдельным экземпляром
редактора, который выполняет валидацию и применение значения.
Пользователь выбирает поле по номеру и вводит новое значение — редактируется
только изменённое поле.

Предусмотрены редакторы для строковых полей (`StringFieldEditor`) и дат
(`DateFieldEditor`). Паттерн расширяем: для нового типа данных достаточно
создать новый класс-наследник `FieldEditor`.

Мотивация: [NFR01](01_Introduction_and_Goals.md#13-цели-качества) — пошаговое
редактирование одного поля проще для пользователя, чем повторный ввод всех данных.
[NFR04](01_Introduction_and_Goals.md#13-цели-качества) — добавление нового поля
требует только создания нового редактора.
